# 游客登录与完善资料弹窗修复方案

> 目标：修复“游客点击『已有账号』退出到登录页后，再点首页重新进入游客状态，APP 下载不再弹出『完善个人信息』弹窗”的问题。  
> 适用范围：`HomeView` 的 KeepAlive 恢复、游客登录状态、APP 下载入口、`GuestTipModal` 行为。  
> 注意：当前工作区已有 `SearchBar.vue`、`ProfileView.vue`、`GuestTipModal.vue` 的未提交改动。实施时必须在其基础上增量修改，不得覆盖或还原已有改动。

## 根因

`App.vue` 用 `KeepAlive` 缓存 `HomeView`：

```vue
<KeepAlive include="HomeView">
  <component :is="Component" :key="route.name === 'home' ? 'home' : route.fullPath" />
</KeepAlive>
```

首次进入首页时，`HomeView.vue` 的 `onMounted` 会调用 `performTouristLogin()`。但游客在完善资料弹窗中点击“已有账号”后，`userLogout()` 清空本地会话并跳转 `/login`；再由登录页点击“首页”回到 `/` 时，首页实例不会重新挂载，只会触发 `onActivated`。

当前 `onActivated` 只恢复首页列表、滚动位置和 session 缓存，**没有调用 `performTouristLogin()`**。此时：

1. `userStore.profile` 已在登出时被清空；
2. `userStore.isGuest` 为 `false`；
3. `SearchBar.vue` 的 `handleAppDownload()` 因而直接打开下载链接，不显示 `GuestTipModal`。

## 目标行为

| 场景 | 期望结果 |
| --- | --- |
| 首次进入首页且无会话 | 自动游客登录，APP 下载显示完善资料弹窗。 |
| 游客点击“已有账号” | 退出本地/远端会话，跳转登录页。 |
| 登录页点击首页 | 首页恢复前重新建立游客会话。游客会话成功后，APP 下载再次显示完善资料弹窗。 |
| 正式账号登录后进入首页 | 不覆盖正式用户会话；APP 下载保持直接下载。 |
| 用户在游客会话恢复中点击 APP 下载 | 不打开外链，显示“正在恢复游客身份，请稍后再试”。 |
| 用户关闭弹窗遮罩或右上角 X | 仅关闭弹窗，继续保留游客会话；不退出登录。 |

## 实施步骤

### 1. 在首页恢复激活时补建游客会话

修改 `src/views/HomeView.vue`。

#### 1.1 新增状态和 store

```ts
import { ref, onMounted, onBeforeUnmount, computed, onActivated } from 'vue'
import { useUserStore } from '@/stores/user'

type GuestSessionState = 'loading' | 'ready' | 'failed'

const userStore = useUserStore()
const guestSessionState = ref<GuestSessionState>('ready')
let isRestoringGuestSession = false
```

#### 1.2 新增会话恢复函数

```ts
const ensureTouristSessionOnActivated = async () => {
  if (isRestoringGuestSession) return

  // 先与 localStorage 同步，避免根据过期的 Pinia 内存状态决策。
  userStore.hydrateFromStorage()

  // 正式用户或当前有效游客均不应被新的游客登录覆盖。
  if (userStore.profile) {
    guestSessionState.value = 'ready'
    return
  }

  isRestoringGuestSession = true
  guestSessionState.value = 'loading'

  try {
    await performTouristLogin({
      route,
      silentLoading: true,
      silentSuccessToast: true,
    })

    // touristLogin 成功后会写 localStorage；再次 hydrate 才能保证 Pinia 已同步。
    userStore.hydrateFromStorage()
    guestSessionState.value = userStore.profile ? 'ready' : 'failed'
  } catch {
    // performTouristLogin 当前会自行捕获错误；此处保留防御性处理。
    guestSessionState.value = 'failed'
  } finally {
    isRestoringGuestSession = false
  }
}
```

说明：这里仅在 `profile` 为空时补建游客账号，正好覆盖“点击已有账号后已退出”的场景，不会覆盖已登录正式用户。

#### 1.3 修改 `onActivated`

把现有 `onActivated(() => { ... })` 改为 `onActivated(async () => { ... })`。

- 保留 `isFirstActivation` 的首次直接返回逻辑，避免和 `onMounted` 的首次 `performTouristLogin()` 发出重复请求。
- 在**非首次激活**分支最前面执行：

```ts
await ensureTouristSessionOnActivated()
```

- 会话恢复结束后，再执行原来的 session 恢复、列表缓存恢复、广告延迟加载、滚动位置恢复逻辑。

不要删除 `onMounted` 中已有的 `performTouristLogin({ route })`；首次进入首页仍走原有流程。

### 2. APP 下载入口等待游客会话恢复

修改 `src/components/SearchBar.vue`。

#### 2.1 接收父组件状态

```ts
interface Props {
  keyword?: string
  guestSessionState?: 'loading' | 'ready' | 'failed'
}

const props = withDefaults(defineProps<Props>(), {
  keyword: '',
  guestSessionState: 'ready',
})
```

在 `HomeView.vue` 中改为：

```vue
<SearchBar
  :keyword="searchKeyword"
  :guest-session-state="guestSessionState"
  @search="handleSearch"
/>
```

#### 2.2 修改 `handleAppDownload`

```ts
const handleAppDownload = () => {
  // 处理从 KeepAlive 恢复后的 store 同步。
  userStore.hydrateFromStorage()

  if (props.guestSessionState === 'loading') {
    showToast('正在恢复游客身份，请稍后再试')
    return
  }

  if (props.guestSessionState === 'failed') {
    showToast('游客身份初始化失败，请刷新后重试')
    return
  }

  if (userStore.isGuest) {
    showGuestTip.value = true
    return
  }

  openAppDownload()
}
```

这里的目的不是仅“再读一次 localStorage”，而是防止游客登录异步请求尚未完成时提前打开下载外链。

### 3. 修正共享弹窗的按钮语义

修改 `src/components/GuestTipModal.vue`。

目前“已有账号”、遮罩点击和右上角 X 共用 `handleClose`，都会退出登录并跳转登录页。这会让用户仅想关闭弹窗时意外退出游客会话。

应拆为两个方法：

```ts
const isSwitchingAccount = ref(false)

const handleDismiss = () => {
  emit('update:visible', false)
  emit('close')
}

const handleExistingAccount = async () => {
  if (isSwitchingAccount.value) return
  isSwitchingAccount.value = true

  try {
    // userLogout 无论远端成功与否都会清理本地会话。
    await userLogout()
  } finally {
    emit('update:visible', false)
    emit('close')
    router.replace('/login')
    isSwitchingAccount.value = false
  }
}
```

模板对应关系：

```vue
<div v-if="visible" @click.self="handleDismiss">
  <div class="guest-tip-close" @click="handleDismiss">...</div>

  <button
    class="guest-tip-btn guest-tip-btn-secondary"
    :disabled="isSwitchingAccount"
    @click="handleExistingAccount"
  >
    {{ isSwitchingAccount ? '正在退出...' : '已有账号' }}
  </button>
</div>
```

`router.replace('/login')` 用于替代 `router.push('/login')`，避免用户按浏览器返回键回到一个已退出的游客页面。

### 4. 给游客登录增加请求去重

修改 `src/composables/useTouristLogin.ts`，避免 `onMounted`、`onActivated` 和快速路由操作同时发起多个游客登录请求。

建议在模块作用域维护：

```ts
let touristLoginInFlight: Promise<void> | null = null
```

实现要求：

1. 调用前先 `userStore.hydrateFromStorage()`。
2. `userStore.profile` 存在时直接返回，不能覆盖有效正式账号。
3. 若 `touristLoginInFlight` 存在，直接 `await` 同一个 Promise。
4. 成功或失败都在 `finally` 中将 `touristLoginInFlight = null`，确保下次可重试。
5. 保留调用方传入的静默 loading/toast 选项；同一请求不应重复弹 toast。

## 验收用例

1. 清理浏览器 Application Storage 的 `user_info`、`user_token`、`token_expire_time`、`isGuest`。
2. 打开首页，等待游客登录完成；点击 APP 下载，应弹出“完善个人信息”。
3. 点击“已有账号”：应跳转 `/login`，并确认上述登录相关 key 已清除。
4. 在登录页点击底部“首页”，等待游客登录完成。
5. 再点击 APP 下载：必须再次弹出“完善个人信息”，不能直接打开下载网址。
6. 重复第 3–5 步至少三次，Network 中每次回到首页最多只有一次游客登录请求。
7. 在完善资料弹窗点击遮罩或右上角 X：弹窗关闭，但仍留在首页；再次点 APP 下载仍会弹出弹窗。
8. 使用正式账号登录后回首页，点击 APP 下载应直接打开下载链接，不能显示游客弹窗。
9. 模拟游客登录失败或在恢复过程中点击 APP 下载：不得打开外链，应显示相应失败/等待提示。
